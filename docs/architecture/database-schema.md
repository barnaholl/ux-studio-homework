# Database Schema

## Overview

SQLite database managed by Prisma ORM. Three tables: `User`, `Contact`, `RefreshToken`. No joins at read time, every relationship is enforced by FK constraints with `ON DELETE CASCADE`.

---

## Tables

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | TEXT | PK | `cuid()` generated by Prisma |
| `email` | TEXT | UNIQUE NOT NULL | Lowercased at registration |
| `passwordHash` | TEXT | NOT NULL | bcrypt, 12 rounds |
| `displayName` | TEXT | NOT NULL | User-chosen display name |
| `phone` | TEXT | nullable | Optional profile field |
| `avatarUrl` | TEXT | nullable | S3 base URL without size suffix |
| `theme` | TEXT | NOT NULL DEFAULT 'system' | `system \| light \| dark` |
| `createdAt` | DATETIME | NOT NULL DEFAULT now | |
| `updatedAt` | DATETIME | NOT NULL | Prisma auto-updates |

### `Contact`

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | TEXT | PK | `cuid()` generated by Prisma |
| `name` | TEXT | nullable | At least one of name/phone/email required (service-layer rule) |
| `phone` | TEXT | nullable | |
| `email` | TEXT | nullable | |
| `sortName` | TEXT | NOT NULL DEFAULT '' | `COALESCE(name, email, phone).toLowerCase()` — pre-computed for index-friendly sorting |
| `avatarUrl` | TEXT | nullable | S3 base URL or static `/avatars/*.png` path |
| `isFavourite` | BOOLEAN | NOT NULL DEFAULT false | See [ADR-001](decisions/ADR-001-ifavourite-column.md) |
| `createdAt` | DATETIME | NOT NULL DEFAULT now | |
| `updatedAt` | DATETIME | NOT NULL | Prisma auto-updates |
| `deletedAt` | DATETIME | nullable | `NULL` = active; non-NULL = soft-deleted, pending purge |
| `userId` | TEXT | FK → User.id CASCADE | Every contact is strictly private to one user |

### `RefreshToken`

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | TEXT | PK | `cuid()` generated by Prisma |
| `tokenHash` | TEXT | UNIQUE NOT NULL | SHA-256 of the raw 48-byte token — raw token is never stored |
| `expiresAt` | DATETIME | NOT NULL | `createdAt + 7 days`; checked on every refresh |
| `createdAt` | DATETIME | NOT NULL DEFAULT now | |
| `userId` | TEXT | FK → User.id CASCADE | Enables "log out all devices" via `deleteMany({ where: { userId } })` |

See [ADR-002](decisions/ADR-002-refresh-token-sql-table.md) for why this is a SQL table rather than a Redis entry.

## Indexes

| Index | Table | Columns | Purpose |
|---|---|---|---|
| `User_email_key` | User | `email` | UNIQUE — fast login lookup |
| `Contact_userId_deletedAt_sortName_idx` | Contact | `(userId, deletedAt, sortName)` | `findAll` name-sort: filters active contacts for a user, ordered by `sortName` |
| `Contact_userId_deletedAt_createdAt_idx` | Contact | `(userId, deletedAt, createdAt)` | `findAll` date-sort: same filter, ordered by `createdAt DESC` |
| `RefreshToken_tokenHash_key` | RefreshToken | `tokenHash` | UNIQUE — O(1) lookup on refresh |
| `RefreshToken_userId_idx` | RefreshToken | `userId` | "Log out all devices" query |

### Why composite indexes on `Contact`?

The `findAll` query always applies three conditions together:

```sql
WHERE userId = ?        -- every user sees only their own contacts
  AND deletedAt IS NULL -- only active contacts (soft-delete)
ORDER BY sortName ASC   -- or createdAt DESC
```

A composite index `(userId, deletedAt, sortName)` lets SQLite satisfy the entire `WHERE + ORDER BY` in a single index scan with no sort step. Without it, SQLite would use the `userId` index alone and then sort in memory — O(n log n) per page instead of O(1) index seek.

The two indexes cover both sort modes (`sort=name` and `sort=createdAt`) without duplication.

---

## Soft-Delete Pattern

Contacts are never immediately hard-deleted. On `DELETE /contacts/:id`:

1. `deletedAt` is set to `NOW()` — the row disappears from all `findAll` / `findOne` queries immediately.
2. A BullMQ job is enqueued with a 10-second delay.
3. The frontend shows an undo toast for 10 seconds. If clicked, `POST /contacts/:id/restore` sets `deletedAt = null`.
4. After the delay, the BullMQ processor hard-deletes the row and its S3 avatar files.

`deletedAt IS NULL` is part of both composite indexes, so soft-deleted rows are never read by normal queries at zero extra cost.
